## 为什么分片？怎么做的？怎么确保分片策略效果更好？

### 为什么分片

大文件一次性上传的问题：
- 网络中断需全部重传
- 内存一次性加载压力大
- 单请求超时限制

分片后支持**断点续传**、**并行上传**、**失败重传单个分片**。

### 怎么做

```
前端 File.slice() 切分 → 并行上传每个分片 → 后端暂存临时目录
→ 所有分片上传完 → 合并为完整文件
```

### 怎么确保分片策略效果更好

- **动态分片大小**：弱网用小分片提高成功率，好网用大分片减少请求数
- **并发控制**：限制同时上传的分片数（3~5 个），避免打满带宽
- **断点续传**：上传前请求已上传分片列表，跳过已完成的分片
- **MD5 校验**：每个分片携带 MD5，后端校验完整性

## 为什么消息队列？分片上传完成后怎么做？

### 为什么用消息队列

合并 + 索引构建 + 向量化是耗时操作，同步处理会导致：
- 前端请求超时
- 用户体验差
- 高并发下服务器阻塞

MQ 方案：异步处理、削峰填谷、消息持久化 + 重试、解耦

### 分片上传完成后完整流程

```
所有分片上传完 → 通知后端合并文件 → 发送 MQ 消息
  → 消费者收到消息 → 文档解析（PDF/Word 转文本）
  → 文本分块（chunk） → 向量化（embedding）
  → 写入向量数据库 → 更新索引状态
```

## 混合检索

### 三种检索方式对比

| 检索方式 | 优点 | 缺点 |
|---|---|---|
| **关键词检索（BM25）** | 精确匹配，对术语友好 | 语义理解差，同义词不匹配 |
| **向量检索（语义）** | 理解语义，能匹配同义表达 | 对冷门/专有名词效果差 |
| **记忆检索** | 利用历史交互上下文 | 依赖记忆的构建质量 |

### 融合策略

```
Score = α × 向量相似度 + β × BM25 分数 + γ × 记忆相关性
```

- α、β、γ 可动态调整（技术文档提高 BM25 权重，开放问答提高向量权重）
- 先用向量检索召回 Top K，再用关键词重排序（Rerank）
- 记忆混合：将用户历史对话中的关键信息检索出来作为上下文注入

### 怎么让记忆混合检索效果更好

- **记忆压缩**：对长历史做摘要提取，而非全量存储
- **分层记忆**：短期（当前会话）→ 长期（用户画像/偏好）
- **时效衰减**：太久远的记忆降权
- **相关性过滤**：检索到的记忆需与当前 query 二次相关性判断，不相关的丢弃

## MySQL 索引 — B+ 树结构，1000 万条记录树高

### B+ 树要点

- 非叶子节点只存**索引键 + 指针**，不存数据
- 叶子节点存**完整记录**，且用**双向链表**串联
- 非叶子节点可存更多键 → 树更低 → IO 次数更少

### 1000 万条树高计算

假设：
- MySQL 页大小：**16KB**
- 索引键（BigInt）：**8B**
- 指针：**6B**
- 非叶子节点每页可存：`16KB / (8B + 6B) ≈ 1170` 个键值对
- 叶子节点每条记录（按 1KB 估算）可存：`16KB / 1KB = 16` 条

**树高计算：**
```
第 1 层（根，非叶子）：     1 页 × 1170 个指针 → 指向 1170 个页
第 2 层（非叶子）：       1170 页 × 1170 个指针 → 指向 ~136.9 万 个页
第 3 层（叶子）：         136.9 万 页 × 16 条/页 → 可存约 2190 万 条
```

**结论：1000 万条数据 → B+ 树高 = 3 层**

也就是说查询 1000 万条记录中的任意一行，**最多 3 次 IO**（2 次非叶子 + 1 次叶子），这就是 MySQL 索引高效的原因。

## vibecoding / agent / openclaw

### Vibe Coding

Andrej Karpathy 提出的概念：通过自然语言描述需求，让 AI 自动生成代码，开发者从"写代码"变成"审代码"。本质是 AI 辅助编程的最高形态。

### Agent（AI Agent）

具备**自主规划 + 工具调用 + 记忆**能力的 AI 系统。典型范式：
`LLM + 工具（Tool） + 规划（Planning） + 记忆（Memory）`

### OpenCLAW

可能指某个开源 Agent 框架或 CLAW 相关项目。如果是 Open-CLAW，可以往 AI Agent 工程化方向回答（如 LangChain、AutoGPT、MetaGPT 等）。

### 个人看法

简单 CRUD 适合 vibe coding，复杂业务逻辑仍需人工设计。Agent 在自动化流程上有价值，但在关键路径上需要 human-in-the-loop。
