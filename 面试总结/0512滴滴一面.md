## 参考回答整理

### 1. 设计模式用到了哪些

项目里可以重点说 **策略模式**、**模板方法模式**、**工厂模式**、**责任链模式** 和 **代理模式**。

- **策略模式**：不同业务类型走不同处理逻辑，用 `Map<String, Handler>` 或 Spring 注入多个实现类，避免大量 `if-else`。
- **模板方法模式**：固定流程抽到父类，比如参数校验 → 业务处理 → 记录日志 → 返回结果，差异点交给子类。
- **工厂模式**：根据类型创建不同处理器，比如支付方式、消息类型、文件解析器。
- **责任链模式**：参数校验、权限校验、风控校验、业务校验按链路依次执行。
- **代理模式**：Spring AOP 本质就是动态代理，常用于日志、事务、权限、监控。

面试里可以说：我项目里最常用的是 **策略模式 + AOP 代理模式**，一个解决业务分支扩展，一个解决横切逻辑复用。

### 2. Java 反射，反射底层怎么实现

**反射** 就是程序在运行时获取类的信息，并动态创建对象、调用方法、访问字段。

底层可以这样理解：类被加载后，JVM 会在方法区 / 元空间保存类元数据，同时在堆里生成一个对应的 `Class` 对象。`Class` 对象是访问这些元数据的入口，通过它可以拿到 `Method`、`Field`、`Constructor` 等对象。

调用 `method.invoke()` 时，JDK 会先做访问检查、参数检查，然后通过底层调用机制执行目标方法。早期反射主要走 native 调用，调用次数多后会生成字节码访问器来提升效率。

一句话回答：反射的核心入口是 `Class` 对象，`Class` 对象关联 JVM 中的类元数据，`Method.invoke()` 最终会通过 JVM 的方法调用机制执行目标方法。

### 3. 线程池里的线程名称想自定义，怎么实现

可以自定义 `ThreadFactory`，在线程创建时指定线程名。

```java
ThreadFactory threadFactory = new ThreadFactory() {
    private final AtomicInteger index = new AtomicInteger(1);

    @Override
    public Thread newThread(Runnable r) {
        return new Thread(r, "order-task-pool-" + index.getAndIncrement());
    }
};

ThreadPoolExecutor executor = new ThreadPoolExecutor(
        5,
        10,
        60,
        TimeUnit.SECONDS,
        new LinkedBlockingQueue<>(100),
        threadFactory,
        new ThreadPoolExecutor.AbortPolicy()
);
```

实际项目里也可以用 Guava 的 `ThreadFactoryBuilder`：

```java
ThreadFactory factory = new ThreadFactoryBuilder()
        .setNameFormat("order-task-pool-%d")
        .build();
```

这样做的好处是日志和线上排查时可以快速定位线程池来源。

### 4. 自定义拒绝策略，保证任务最终执行

如果线程池已经达到最大线程数，并且阻塞队列也满了，新任务会触发拒绝策略。默认的 `DiscardPolicy` 会直接丢弃任务，如果任务不能丢，就要自定义 `RejectedExecutionHandler`。

最简单的方式是让提交任务的线程自己执行，类似 `CallerRunsPolicy`：

```java
new RejectedExecutionHandler() {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        if (!executor.isShutdown()) {
            r.run();
        }
    }
}
```

这种方式不会丢任务，但会阻塞调用方，相当于给上游一个反压。

如果任务非常重要，不能因为服务重启或内存队列满而丢，拒绝策略里可以把任务先持久化到 **MQ / Redis / DB**，再由补偿任务异步捞出来执行。

```java
new RejectedExecutionHandler() {
    @Override
    public void rejectedExecution(Runnable r, ThreadPoolExecutor executor) {
        // 1. 记录告警日志
        // 2. 将任务参数序列化后写入 MQ / Redis / DB
        // 3. 后续通过补偿线程或定时任务重新执行
    }
}
```

面试总结：如果只是不想丢，可以用 `CallerRunsPolicy`；如果要求最终一定执行，最好落 MQ / Redis / DB，并且保证任务幂等。

### 5. 不想每个方法手写入参出参日志，怎么做

可以用 **Spring AOP**。把打印入参、出参、耗时、异常这些横切逻辑放到切面里，用 `@Around` 环绕通知统一处理。

```java
@Aspect
@Component
public class LogAspect {

    @Around("execution(* com.xxx.service..*(..))")
    public Object log(ProceedingJoinPoint joinPoint) throws Throwable {
        Object[] args = joinPoint.getArgs();
        long start = System.currentTimeMillis();

        try {
            Object result = joinPoint.proceed();
            long cost = System.currentTimeMillis() - start;
            // 打印方法名、入参、出参、耗时
            return result;
        } catch (Throwable e) {
            // 打印异常日志
            throw e;
        }
    }
}
```

这类日志、事务、权限、限流、监控都属于横切逻辑，适合用 AOP，而不是侵入每个业务方法。

### 6. AOP 切点配的是路径吗？不在一个路径下怎么办

切点不是简单的文件路径，而是 **切点表达式**。常见的 `execution` 表达式匹配的是包名、类名、方法名和参数。

```java
@Pointcut("execution(* com.xxx.controller..*(..))")
public void controllerLog() {}
```

如果多个接口不在同一个包下，可以用 `||` 组合多个表达式：

```java
@Around("execution(* com.xxx.order..*(..)) || execution(* com.xxx.user..*(..))")
```

更推荐的方式是用 **自定义注解** 做切点。这样不管方法在哪个包下，只要加了注解就能被切到。

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface LogRecord {
}
```

```java
@Around("@annotation(com.xxx.LogRecord)")
public Object log(ProceedingJoinPoint joinPoint) throws Throwable {
    return joinPoint.proceed();
}
```

面试回答：如果方法分布在多个包里，我更倾向用注解切点，范围更明确，也不容易误伤其他方法。

### 7. 一条 update 语句的执行过程

以 MySQL InnoDB 为例，一条 `update` 大致这样执行：

1. 客户端发送 SQL 到 MySQL Server。
2. Server 层做连接管理、语法解析、语义检查。
3. 优化器选择执行计划，比如走哪个索引。
4. 执行器调用 InnoDB 存储引擎接口。
5. InnoDB 根据索引找到要更新的记录。
6. 更新前写 **undo log**，用于事务回滚和 MVCC。
7. 修改 Buffer Pool 里的数据页。
8. 写 **redo log**，保证崩溃恢复。
9. Server 层写 **binlog**，用于主从复制和数据恢复。
10. 事务提交时，通过两阶段提交保证 redo log 和 binlog 一致。

一句话：`update` 的核心是先定位数据，再写 undo log，修改 Buffer Pool，写 redo log 和 binlog，最后提交事务。

### 8. explain 主要看哪些字段

使用 `explain` 分析 SQL 时，重点看这些字段：

| 字段 | 关注点 |
|---|---|
| `type` | 访问类型，最好是 `ref`、`range`，避免 `ALL` |
| `possible_keys` | 理论上可能使用的索引 |
| `key` | 实际使用的索引 |
| `key_len` | 实际用到的索引长度，可判断联合索引用了几列 |
| `rows` | 预估扫描行数，越小越好 |
| `Extra` | 是否出现 `Using filesort`、`Using temporary` |
| `filtered` | 过滤比例 |
| `ref` | 哪个字段或常量和索引比较 |

面试里可以说：我一般重点看 `type`、`key`、`rows` 和 `Extra`。如果 `type` 是 `ALL`，`rows` 很大，或者 `Extra` 出现 `Using filesort` / `Using temporary`，就要重点优化。

### 9. SQL 走了索引，但不是预期索引，怎么指定

MySQL 可以使用索引提示：

```sql
SELECT *
FROM user FORCE INDEX(idx_name)
WHERE name = 'Tom';
```

常见有三种：

```sql
USE INDEX(idx_name)      -- 建议优化器使用某个索引
FORCE INDEX(idx_name)    -- 更强硬地要求使用某个索引
IGNORE INDEX(idx_name)   -- 忽略某个索引
```

但要补一句：手动指定索引一般是兜底方案，最好先分析统计信息、SQL 写法和索引设计。因为数据分布变化后，强制指定索引可能反而变慢。

### 10. MQ 推模式和拉模式的区别

**推模式** 是 Broker 主动把消息推给消费者。优点是实时性好，消费者不用频繁轮询；缺点是如果消费者处理能力不足，容易被推爆，需要 Broker 做流控。

**拉模式** 是消费者主动从 Broker 拉取消息。优点是消费者可以根据自己的能力控制拉取频率和批量大小，更容易做限流和批量消费；缺点是可能存在轮询延迟，拉太频繁也会浪费资源。

总结：推模式实时性更好，但对消费者压力更大；拉模式消费者自主性更强，更适合做流控和批量消费。

---

### 11. 推拉模式在技术选型上的影响

推模式更偏 **Broker 主导**。生产者只负责把消息发到 Broker，影响不大；Broker 需要主动分发消息，如果消费者处理不过来，就要做限流、重试和消费者保护。对消费者来说，它是被动接收，实时性强，但压力也更不可控。

拉模式更偏 **消费者主导**。生产者同样只负责发送消息，消息堆积主要留在 Broker；消费者自己控制拉取频率、批量大小和消费节奏，所以更容易根据自身负载做流控。但消费者要处理 offset 管理、空轮询、长轮询等问题。

面试里可以补充：Kafka、RocketMQ 本质上更偏拉模式，很多所谓的推，其实是基于长轮询封装出来的。

| 对比 | 抽象类 | 接口 |

### 13. ACK 返回时网络挂了怎么办

这个问题的核心是：业务可能已经处理成功，但 ACK 没有成功发到 Broker，Broker 会认为消费失败，然后重新投递这条消息。

所以消费者不能假设消息只会被消费一次，而是要保证 **幂等消费**。

常见处理方式：

1. 每条消息有唯一 `messageId` 或业务唯一键。
2. 消费前查 DB / Redis，看这条消息是否已经处理过。
3. 处理时记录状态，比如 `处理中`、`处理成功`、`处理失败`。
4. 如果重复投递时发现已经成功，就直接返回 ACK。
5. 如果长时间停留在 `处理中`，用超时时间和定时任务做补偿。

状态表可以类似这样：

| message_id | status | update_time |
|---|---|---|
| xxx | processing | 2026-05-12 |
| yyy | success | 2026-05-12 |

面试回答：ACK 丢失导致重复投递是 MQ 的常见情况，所以消费者要做幂等。可以用消息表记录处理中和成功状态，配合唯一索引、事务和定时补偿，保证最终一致。

### 14. 为什么使用双层 Token

双层 Token 一般指：

- **Access Token**：短期有效，用来访问接口。
- **Refresh Token**：长期有效，用来刷新 Access Token。

这样设计主要是为了兼顾安全性和用户体验。

1. **降低泄露风险**：Access Token 有效期短，即使泄露，攻击窗口也比较小。
2. **提升用户体验**：Access Token 过期后，可以用 Refresh Token 换新的 Access Token，不需要用户频繁登录。
3. **便于权限控制**：服务端可以保存 Refresh Token 状态，实现主动失效、踢下线、单设备登录等能力。
4. **减少账号密码暴露**：不需要每次 Token 过期都重新输入账号密码。

一句话：Access Token 负责接口访问，生命周期短；Refresh Token 负责续期，生命周期长。这样既安全，又不会让用户频繁登录。
