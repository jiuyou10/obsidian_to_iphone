# 执行 SELECT 语句的完整流程

MySQL 的架构共分为两层：**Server 层** 和 **存储引擎层**。

Server 层负责连接管理、解析、优化、执行；存储引擎层负责数据的存取（InnoDB、MyISAM 等）。

---

## 第一步：连接器

### 连接过程

1. **TCP 三次握手** — MySQL 基于 TCP 协议传输
2. **验证身份** — 检验用户名和密码
3. **获取权限** — 从权限表中读取用户的权限列表，存入会话缓存
4. **后续操作** — 该连接内的所有 SQL 操作都基于**连接时缓存的权限**做校验

> 注意：权限是连接时读取的，即使之后管理员改了权限，当前已存在的连接也不受影响，需重连才能生效。

### 常用命令

```sql
SHOW PROCESSLIST;    -- 查看当前所有连接状态
```

### 常见问题

**Q：空闲连接会一直占用吗？**
不会。MySQL 通过 `wait_timeout` 参数控制最大空闲时长（默认 8 小时），超时后自动断开。

**Q：连接数有限制吗？**
MySQL 服务支持的最大连接数由 `max_connections` 参数控制，默认值通常为 151。

**Q：短连接 vs 长连接？**

| 类型 | 特点 | 优点 | 缺点 |
|------|------|------|------|
| 短连接 | 每次查询都新建、断开连接 | 内存占用低 | 频繁建立连接开销大 |
| 长连接 | 一条连接复用多次查询 | 减少 TCP 握手和认证开销 | 内存占用可能过大 |

推荐使用**长连接**，但需解决内存占用过大的问题：

- **方案一**：定期断开长连接，释放内存
- **方案二**：客户端主动重置连接
  ```sql
  mysql_reset_connection();  -- MySQL 5.7+，不重连但重置会话状态
  ```

---

## 第二步：查询缓存（MySQL 8.0 已移除）

### 历史机制

- 执行 SELECT 前，先查缓存（key-value 结构，key 为 SQL 语句）
- 命中则直接返回结果，跳过后续所有步骤
- 未命中则执行完整流程，结果存入缓存

### 为什么被移除？

- **缓存失效太频繁**：只要表有更新操作（INSERT/UPDATE/DELETE），该表的所有查询缓存全部清空
- 对于写密集的应用，缓存命中率极低，反而带来维护开销
- MySQL 8.0 **完全移除了查询缓存功能**

> 现在可以用 Redis 等外部缓存替代。

---

## 第三步：解析 SQL（Parser）

### 词法分析

扫描 SQL 语句，识别出关键字、表名、字段名、运算符等**词法单元**（Token）。

### 语法分析

- 根据词法分析的结果，判断 SQL 是否符合 MySQL 语法规则
- 若语法错误，返回 `You have an error in your SQL syntax`
- 若语法正确，构建**语法树**
- 语法树记录了 SQL 的类型（SELECT/INSERT/UPDATE）、涉及的表名、字段名、WHERE 条件等结构信息

---

## 第四步：执行 SQL

执行阶段又分为 **预处理 → 优化器 → 执行器** 三个子阶段。

### 4.1 预处理

检查语法树中的表和字段是否真实存在：

- 检查 SQL 中的表名是否存在于数据库中
- 检查每个字段名是否存在于对应的表中（解决歧义）
- 将 `SELECT *` 的 `*` 扩展为表上的**所有列名列表**
- 解析表别名和字段别名

### 4.2 优化器

**职责**：确定 SQL 语句的执行方案（选择索引、决定表连接顺序等）。

- 在有多个索引可选时，基于**查询成本**（I/O 次数、CPU 消耗）选择成本最低的方案
- 分析 WHERE 条件，判断哪些索引可用，以及如何组合使用
- 对于多表 JOIN，决定表的访问顺序

```sql
EXPLAIN SELECT * FROM product WHERE id = 1;
-- 用 EXPLAIN 查看优化器选择的索引和执行计划

-- 更详细的分析方式：
EXPLAIN FORMAT=JSON ...;

-- 开启优化器追踪：
SET optimizer_trace = 'enabled=on';
```

### 4.3 执行器

执行器根据优化器生成的执行计划，调用存储引擎的接口来读写数据。

> 以下以 InnoDB 为例。

#### 4.3.1 主键索引查询

```sql
SELECT * FROM product WHERE id = 1;
```

- 条件 `id = 1` 通过主键索引（聚簇索引）定位
- InnoDB 在 id 的 B+Tree 中直接搜索到叶子节点，取出整行数据
- 因为主键索引的叶子节点直接包含完整行数据，**无需回表**

#### 4.3.2 全表扫描

```sql
SELECT * FROM product WHERE name = 'phone';
```

以下情况可能触发全表扫描：

- 没有合适的索引可用
- 优化器评估后发现走索引的成本反而高于全表扫描（如需要回表的行数过多）
- 查询需要读取表中大部分数据

全表扫描即**顺序读**主键索引的 B+Tree 所有叶子节点，逐行检查是否满足 WHERE 条件。

> 注意：所有叶子节点的页通过双向链表连接，物理上连续存储（区）；非叶子节点页不连续。InnoDB 全表扫描，本质就是**从聚簇索引叶子链表头部开始，顺着双向链表挨个遍历所有叶子页**。

---

#### 4.3.3 索引下推（Index Condition Pushdown, ICP）

**解决的问题**：在使用**二级索引**查询时，把部分 WHERE 条件"下推"到存储引擎层提前过滤，减少回表次数。

> 注意：不使用 ICP 时，只有最左前缀字段在索引层筛选，其余条件都要 **回表后** 在 Server 层判断。

**不使用 ICP 的过程**：
```
二级索引按最左前缀筛选出满足索引条件的行 → 回表读取完整行 → Server 层判断剩余条件
```

**使用 ICP 的过程**：
```
二级索引按最左前缀筛选，同时把剩余条件下推到索引层面过滤 → 只对满足全部条件的行回表
```

**举例**：

```sql
-- 联合索引 (name, age)
SELECT * FROM t WHERE name = '张' AND age = 18;
```

- `name = '张'` 走索引前缀匹配，定位到二级索引的连续范围
- **无 ICP**：扫到 `name = '张'` 的每条索引记录 → 立即回表 → Server 层判断 `age = 18`
- **有 ICP**：扫索引时，直接在索引记录的 `age` 字段上判断 `age = 18` → 只对 `age = 18` 的行回表

**核心收益**：把过滤动作提前到**索引扫描阶段**，避免不必要的回表 I/O。

ICP 由 MySQL 5.6 引入，默认开启：

```sql
SET optimizer_switch = 'index_condition_pushdown=on';  -- 默认 on
```

---

## 总结流程图

```
客户端发送 SQL
     ↓
① 连接器 — TCP 握手 + 认证 + 权限缓存
     ↓
② 查询缓存 — 8.0 已移除（略过）
     ↓
③ 解析器 — 词法分析 → 语法分析 → 语法树
     ↓
④ 执行器
     ├─ 4.1 预处理 — 检查表/字段是否存在，展开 *
     ├─ 4.2 优化器 — 选择索引和执行计划（EXPLAIN 查看）
     └─ 4.3 执行器 — 调用存储引擎读写数据
          ├─ 主键索引查询（聚簇索引，无需回表）
          ├─ 全表扫描（顺序扫主键索引叶子节点）
          └─ 索引下推 ICP（二级索引提前过滤，减少回表）
     ↓
返回结果给客户端
```

# MySQL 表中一行记录是怎么存储的

## 表空间结构层级

```
表空间 (Tablespace)
  └─ 段 (Segment)
       └─ 区 (Extent)  — 64 个连续页 = 1MB
            └─ 页 (Page)  — 16KB，磁盘 I/O 最小单元
                 └─ 行 (Row)
```

### 页（Page）

- **InnoDB 以页为单位读写数据**，默认每页 **16KB**
- 页是 InnoDB 磁盘管理和 I/O 的**最小单元**
- 常见页类型：数据页（B+Tree 节点）、Undo 页、系统页、事务数据页等

### 区（Extent）

- 由 **64 个连续的页**组成，大小固定为 **1MB**（64 × 16KB）
- 当表中数据量增大、为索引分配空间时，**不再按页分配，而是按区分配**
- 目的是让 B+Tree 叶子节点链表中**相邻的页在物理上也相邻**，从而利用**顺序 I/O** 提升扫描性能

### 段（Segment）

- 表空间由各个段组成，段由多个区组成
- **数据段**（Data Segment）：B+Tree 的**叶子节点**，存放实际数据
- **索引段**（Index Segment）：B+Tree 的**非叶子节点**，存放索引目录
- **回滚段**（Rollback Segment）：存放 Undo Log，用于 MVCC

---

## InnoDB 行格式（Row Format）

InnoDB 支持 4 种行格式：

| 行格式 | 特点 | 适用场景 |
|--------|------|---------|
| **COMPACT** | 5.0 引入，紧凑存储 | 早期版本默认 |
| **REDUNDANT** | 5.0 之前的老格式 | 兼容旧版 |
| **DYNAMIC** | COMPACT 的增强版 | **MySQL 5.6+ 默认** |
| **COMPRESSED** | DYNAMIC + 页级压缩 | 节省磁盘空间，但增加 CPU 开销 |

> 当前默认行格式为 **DYNAMIC**。

### 行的内部结构（以 COMPACT / DYNAMIC 为例）

一条记录在磁盘上分为两部分：

```
| 额外信息 (Extra Info) | 真实列数据 (Row Data) |
```

#### 额外信息包括

**① 变长字段长度列表**
- 记录每个变长字段（VARCHAR / VARBINARY / TEXT / BLOB 等）的**实际长度**
- 按列**逆序**存放
- 例如 `VARCHAR(10)` 存了 "abc"，则列表中记录长度 3

**② NULL 值列表**
- 用**位图**标记哪些列允许 NULL 且当前为 NULL（详见下文）

**③ 记录头信息**
- 固定 **5 字节**（40 位），包含：
  - `delete_mask`：该记录是否被删除（1 bit）
  - `min_rec_mask`：B+Tree 非叶子节点的最小记录标记
  - `n_owned`：当前记录拥有的记录数（用于 Slot）
  - `heap_no`：当前记录在堆中的序号
  - `record_type`：0 = 普通记录, 1 = 非叶子节点, 2 = Infimum, 3 = Supremum
  - `next_record`：指向下一条记录的**相对偏移量**（构成单向链表）

#### 真实列数据

- 按 CREATE TABLE 时定义的**列顺序**依次存放
- 每列的数据长度由字段类型决定（定长）或从变长字段长度列表读取（变长）
- 每行还有一个 **6 字节的隐藏主键 `DB_ROW_ID`**（如果没有显式定义主键）

> 列数据按定义顺序存放，而变长字段长度列表和 NULL 值列表按**逆序**存放。为什么逆序？因为每条记录的指针（`next_record`）放在头信息中，指向下一条记录。逆序排列可以保证读取记录时，按顺序从左到右解析先得到第一列的信息。

```
记录在页中的存储示意：
[len3][len2][len1] | [head / NULL bitmap] | [col1][col2][col3]
                      ↑
                  指针位置（next_record）
```

## NULL 值是怎么存放的？

### 核心：用位图存，不用实际字节

- 每一列允许 NULL 的，在 NULL 值列表中占 **1 位**
- 如果该列当前为 NULL，对应位标记为 1，否则为 0
- **逆序存放**（第一个位对应最后一列）
- **如果所有列都定义为 NOT NULL，则没有 NULL 值列表**
- 位图按字节对齐：8 个字段用 1 字节，9 个字段用 2 字节

### 例子

```sql
CREATE TABLE t (
    a INT NOT NULL,
    b VARCHAR(10) NOT NULL,
    c INT,
    d VARCHAR(10)
);
```

`c` 和 `d` 允许 NULL，NULL 值列表用 **2 位**表示：

```
位图: [d的标记位] [c的标记位]   ← 逆序
例如 c = NULL, d = 'abc' → 位图 = 0 1
```

**优点**：用 1 位代替可能占用大量空间的 NULL 值，极大节省存储。

## 行溢出后怎么处理？

### 什么是行溢出

InnoDB 一页 16KB，但一行数据可能超过 16KB（如 TEXT / BLOB / 超长 VARCHAR）。

> VARCHAR 最大长度：`65535 字节`，远超一页容量。

### 不同行格式的处理方式

| 行格式 | 处理方式 |
|--------|---------|
| **COMPACT / REDUNDANT** | 前 **768 字节**存放在数据页中，剩余数据写入**溢出页**（Overflow Page），并在原位置记录 **20 字节的指针** |
| **DYNAMIC / COMPRESSED** | **数据页中只存 20 字节指针**，**全部数据**存放在溢出页中（MySQL 5.6+ 默认，更节省主数据页空间） |

### 示意图

```
COMPACT:  [ 前 768B 数据 | 20B 指针 ] → Overflow Page（剩余数据）
DYNAMIC:  [ 20B 指针 ]               → Overflow Page（全部数据）
```

DYNAMIC 格式将大字段完全移出主数据页，让一页能容纳更多行，**减少 B+Tree 的深度**，提升查询效率。

---

## 总结

| 概念 | 要点 |
|------|------|
| 页 | 16KB，磁盘 I/O 最小单元 |
| 区 | 64 页 = 1MB，保证物理连续性以利用顺序 I/O |
| 段 | 数据段（叶子）、索引段（非叶子）、回滚段 |
| 行格式 | DYNAMIC（默认），COMPACT / REDUNDANT / COMPRESSED |
| NULL 存储 | 位图标记，每列 1 位，不占实际存储空间 |
| 行溢出 | COMPACT 存前 768B；DYNAMIC 全放溢出页，主页只留指针 |

# 索引

MySQL 索引可以从四个维度分类。

## 索引分类
### 按数据结构

| 类型            | 特点                | 适用场景                     |
| ------------- | ----------------- | ------------------------ |
| **B+Tree**    | 有序、自平衡、支持范围查询     | InnoDB 默认，几乎所有场景         |
| **Hash**      | O(1) 等值查询，不支持范围查询 | 仅 Memory 引擎              |
| **Full-text** | 全文检索，倒排索引         | 对大文本做模糊匹配（MATCH AGAINST） |

### 按物理存储

| 类型 | 说明 |
|------|------|
| **聚簇索引（主键索引）** | 叶子节点直接存储**完整行数据** |
| **二级索引（辅助索引）** | 叶子节点存储**主键值**，需回表 |

### 按字段特性

| 类型 | 要求 |
|------|------|
| 主键索引 | 唯一 + NOT NULL，一张表最多一个 |
| 唯一索引 | 唯一，允许 NULL |
| 普通索引 | 无特殊要求 |
| 前缀索引 | 只对字符串的前 N 个字符建索引 |

### 按字段个数

| 类型 | 说明 |
|------|------|
| 单列索引 | 一个字段建索引 |
| **联合索引** | 多个字段组合，遵循最左匹配原则 |

---

## 聚簇索引的选择逻辑

InnoDB 按以下优先级选择聚簇索引：

1. 有主键 → 主键作为聚簇索引
2. 无主键 → 第一个 **NOT NULL 唯一列**
3. 都没有 → InnoDB 自动生成 **6 字节隐式自增 id（DB_ROW_ID）**

> 其他索引都是**二级索引**，叶子节点存主键值，需要回表。

---

## 为什么 MySQL 选择 B+Tree？

### vs B 树

| | B 树 | B+ 树 |
|--|------|-------|
| 非叶子节点 | 存 key + data | **只存 key**，一页能存上千个键值 |
| 叶子节点 | 分散数据 | **集中所有数据** + 双向链表连接 |
| 范围查询 | 中序遍历，逐层回溯 | 定位起点后**顺序遍历链表** |
| 查询稳定性 | 不稳定（根节点 1 次 I/O，叶子需多次） | **稳定 O(logN)**（路径长度固定） |

**三点核心差异**：
1. 非叶子节点不存数据 → 一页容纳更多 key → **树更矮，I/O 更少**
2. 叶子节点双向链表 → **范围查询无回溯开销**
3. 所有数据都在叶子 → **查询时间稳定**

### vs 跳表（Skip List）

跳表本质上还是链表，数据量极大时层数会很高，导致**多次 I/O**。而 B+Tree 的层数始终在 3~4 层左右。

### vs Hash

Hash 表 O(1) 等值查询极快，但**不支持范围查询**（`> < BETWEEN ORDER BY`），且哈希冲突时性能下降。

---

## 范围查询流程

```
WHERE id > 10 AND id < 100
```

1. 从 B+Tree 根节点沿索引路径找到第一个满足 `id > 10` 的叶子节点
2. 从该节点开始，顺着叶子节点的**双向链表**向右遍历
3. 直到遇到 `id >= 100` 的节点，停止
注意如果是找id<100的，单一条件
会先从B+树找到第一个>=100的叶子节点，停止，然后往左遍历
整个过程**无需回溯**，顺序 I/O。

---

## 联合索引与最左匹配原则

### 基础规则

联合索引 `(a, b, c)` 相当于按 `a → b → c` 排序：

| 条件 | 是否走索引 | 说明 |
|------|-----------|------|
| `WHERE a=1` | ✅ 全走 | 第一列 |
| `WHERE a=1 AND b=2` | ✅ 全走 | |
| `WHERE a=1 AND b=2 AND c=3` | ✅ 全走 | |
| `WHERE b=2` | ❌ 不走 | 跳过了 a |
| `WHERE a=1 AND c=3` | ⚠️ a 走索引，c 不走 | 跳过了 b |

> **通俗理解**：联合索引像电话簿，先按姓排、再按名排。跳过姓直接查名是查不到的。

### 范围查询的影响

- **前一个字段等值** → 后一个字段还能继续走索引 ✅
- **前一个字段范围查询** → 后一个字段**无法再走索引定位**，只能做过滤 ❌

```sql
WHERE a = 2 AND b = 1     → a、b 都走索引 ✅
WHERE a > 2 AND b = 1     → 只 a 走索引，b 无法定位 ❌

-- a>=2 会被拆成 a>2 和 a=2 两部分
WHERE a >= 2 AND b = 1
```

**为什么范围查询后面的列会失效？**
找到第一个 `a > 2` 的叶子节点后，向右**顺序扫描链表**，此时 `b` 的值在扫描范围内**不是有序的**。

> `>=`、`<=`、`BETWEEN`、`LIKE 'abc%'`（前缀匹配）**不会停止匹配**，因为等值部分索引仍有序。

---

## 索引优化

### 前缀索引优化
- ✅ 减少索引体积，一页能存更多键值，提高查询效率
- ❌ `ORDER BY` 无法使用；无法用作覆盖索引

### 覆盖索引优化

**定义**：查询的所有字段都包含在索引中，无需回表。

```sql
-- 联合索引 (name, age)
SELECT name, age FROM t WHERE name = '张';  -- 无需回表 ✅
SELECT * FROM t WHERE name = '张';           -- 需要回表 ❌
```

### 主键索引自增
- ✅ **自增主键**：追加写入，不会移动数据，页写完自动新建
- ❌ **非自增（如 UUID）**：插入位置随机，可能触发**页分裂**，产生碎片
- 主键字段**越短**，二级索引叶子节点越小，占用空间越小

### 索引列设为 NOT NULL
- NULL 会使优化器选择更复杂
- NULL 在索引中仍占物理位置

---

## 什么时候需要 / 不需要创建索引

### 适合创建
- 经常出现在 WHERE、JOIN、ORDER BY、GROUP BY 中的列
- 区分度高的列（身份证号、手机号）
- 数据量大且查询频繁的表

### 不适合创建
- 数据量小的表（全表扫描更快）
- 频繁增删改的列（维护索引成本高）
- 区分度低的列（如性别、状态码）

## 索引失效场景

| # | 失效场景 | 错误示例 | 正确写法 | 核心原因 |
|---|---------|---------|---------|---------|
| 1 | 模糊匹配前导通配符 | `LIKE '%abc'` / `LIKE '%abc%'` | `LIKE 'abc%'` | 前缀不确定，无法定位范围 |
| 2 | 对索引列做运算/函数 | `YEAR(create_time)=2023` | `create_time >= '2023-01-01'` | 计算破坏有序性 |
| 3 | 隐式类型转换 | `phone = 1380000`（字段 VARCHAR） | `phone = '1380000'` | `CAST(col AS INT)` = 对列做函数 |
| 4 | 违反最左前缀 | `(a,b,c)` 查 `WHERE b=1` | `WHERE a=1 AND b=1` | 跳过第一列 |
| 5 | OR 连接非索引列 | `WHERE a=1 OR b=1`（b 无索引） | `UNION ALL` 或给 b 加索引 | 任意条件需全表扫，整体放弃索引 |

> 注：第 4 条中，范围查询（`>`、`<`）后的列也会失效，但 `>=`、`<=`、`BETWEEN`、`LIKE 'abc%'` 不会。

## MySQL 分页性能分析

### LIMIT 的执行机制

```sql
SELECT * FROM page ORDER BY id LIMIT offset, size;
```

MySQL 执行 `LIMIT offset, size` 的流程是：**先读取 offset + size 行，再丢弃前 offset 行**，而不是直接跳到第 offset 行开始读。

| 示例 | 实际读取行数 | 行为 |
|------|------------|------|
| `LIMIT 0, 10` | 10 行 | 读 10 行，丢弃 0 行 ✅ 高效 |
| `LIMIT 10000, 10` | 10010 行 | 读 10010 行，丢弃前 10000 行 ❌ 大量浪费 |

### 为什么大 offset 慢

随着 offset 增大，MySQL 需要读取越来越多的"无用行"再丢弃，I/O 和 CPU 都线性增长。

不同索引类型的行为：

| 索引类型 | 查询过程 | 特点 |
|---------|---------|------|
| **主键索引（聚簇索引）** | B+Tree 遍历叶子节点链表，读取 offset+size 条完整行，丢弃前 offset 条 | 叶子节点存完整行数据，读取量大 |
| **二级索引** | 先在二级索引读取 offset+size 个主键值，再逐条回表获取完整行，丢弃前 offset 条 | 回表次数多，**offset 过大时可能退化为全表扫描** |

### 优化方案

#### 方案一：子查询优化（推荐）

核心思路：**先在索引中定位起始点，再获取完整数据**，避免扫描 offset 之前的无用行。

**按主键排序的分页：**
```sql
SELECT * FROM page 
WHERE id >= (SELECT id FROM page ORDER BY id LIMIT 10000, 1) 
ORDER BY id LIMIT 10;
```

子查询只用**索引覆盖扫描**（无需回表）找到第 10001 行的 id，外层查询从该 id 开始顺序取 10 行。

**按非主键排序的分页：**
```sql
SELECT * FROM page t1, 
  (SELECT id FROM page ORDER BY user_name LIMIT 6000000, 100) t2  
WHERE t1.id = t2.id;
```

子查询扫描二级索引 `(user_name)` 获取 id（无需回表），外层 JOIN 通过主键回表取完整行。

#### 方案二：游标分页 / Keyset Pagination（深度分页）

**核心思想**：不依赖 offset，而是记录**上一页最后一条数据的 id**，下一页直接从此 id 之后开始取。

```sql
-- 第一页：取前 10 条
SELECT * FROM page ORDER BY id LIMIT 10;

-- 第二页：记录上一页最后一条的 id = 10
SELECT * FROM page WHERE id > 10 ORDER BY id LIMIT 10;

-- 第 N 页：记录上一页最后一条的 id = 10000
SELECT * FROM page WHERE id > 10000 ORDER BY id LIMIT 10;
```

**优点**：
- 无论多深的页码，每次只读需要的 10 行，**性能恒定**
- 无需 offset，不产生"丢弃行"的开销
- 适合海量数据下的**无限滚动**场景

**缺点**：
- 无法直接跳转到任意页码（如"跳到第 1000 页"）
- 排序字段必须唯一且有序（通常用自增主键）
- 当排序条件复杂时（如多字段排序），需要额外维护游标条件

### 总结

| 分页方式 | 适用场景 | 优点 | 缺点 |
|---------|---------|------|------|
| 普通 `LIMIT offset, size` | 小数据量、浅分页 | 写法简单 | offset 越大越慢 |
| 子查询优化 | 深分页、固定排序 | 避免扫描无用行 | 排序字段需建索引 |
| 游标分页（Keyset） | 海量数据、无限滚动 | 性能恒定 O(size) | 不支持随机跳页 |

## 事务

### ACID 特性

| 特性 | 说明 | 实现机制 |
|------|------|---------|
| **原子性 Atomicity** | 事务中的所有操作要么全部完成，要么全部回滚 | **Undo Log**：回滚时利用 undo log 撤销已执行的操作 |
| **一致性 Consistency** | 事务前后数据满足完整性约束，数据库保持合法状态 | 由原子性、隔离性、持久性共同保证 + 业务层约束 |
| **隔离性 Isolation** | 并发事务之间相互隔离，不互相干扰 | **MVCC** + **锁机制** |
| **持久性 Durability** | 事务提交后，对数据的修改永久保存 | **Redo Log**：系统故障后通过 redo log 重放恢复 |

### 并发问题

| 问题 | 定义 | 示意图 |
|------|------|--------|
| **脏读** | 事务 A **读到**了事务 B **未提交**的数据（B 可能回滚，数据无效） | A 读 → B 写（未提交）→ B 回滚 → A 读到的数据已失效 |
| **不可重复读** | 同一事务内**两次读取同一行数据**，结果不一样（另一事务修改并提交了该行） | A 读 → B 改并提交 → A 再读 → 前后不一致 |
| **幻读** | 同一事务内**两次查询同一范围**，记录数量不一样（另一事务插入/删除了新行） | A 查 count → B 插入新行并提交 → A 再查 count → 数量变多 |

### 隔离级别

| 隔离级别                        | 脏读  | 不可重复读 |    幻读    | 实现方式                             |
| --------------------------- | :-: | :---: | :------: | -------------------------------- |
| **读未提交** (Read Uncommitted) |  ❌  |   ❌   |    ❌     | 事务修改直接读，不加锁                      |
| **读提交** (Read Committed)    |  ✅  |   ❌   |    ❌     | **每次** SELECT 都生成新 ReadView；写锁   |
| **可重复读** (Repeatable Read)  |  ✅  |   ✅   | ⚠️（下文详述） | **事务内**复用第一个 ReadView；MVCC + 间隙锁 |
| **串行化** (Serializable)      |  ✅  |   ✅   |    ✅     | 所有读写都加锁，严格串行                     |

> MySQL InnoDB 引擎**默认隔离级别为可重复读**。

---

## MVCC（多版本并发控制）

MVCC 通过**版本链（Undo Log）** 和 **一致性视图（ReadView）**，让事务在不加锁的情况下实现隔离，核心解决**读-写并发冲突**（写不阻塞读，读不阻塞写）。

### 隐藏列

InnoDB 聚簇索引记录中隐藏了两个关键字段：

| 隐藏列 | 作用 |
|--------|------|
| `trx_id` | 最近修改该记录的事务 ID |
| `roll_pointer` | 指向 Undo Log 中该记录的**上一个版本**，所有版本通过该指针形成**版本链** |

```
当前记录 (trx_id=50) → 旧版本 (trx_id=45) → 更旧版本 (trx_id=40) → ...
    ↑
 roll_pointer
```

### ReadView（一致性视图）

ReadView 在事务执行查询时生成，记录了当前数据库的快照状态。

#### 四个核心字段

| 字段 | 说明 |
|------|------|
| `m_ids` | 创建 ReadView 时，当前所有**活跃事务（未提交）**的 ID 列表 |
| `min_trx_id` | `m_ids` 中最小的事务 ID |
| `max_trx_id` | 下一个待分配的事务 ID（当前最大事务 ID + 1） |
| `creator_trx_id` | **创建该 ReadView 的事务**自身的事务 ID |

#### 可见性判断规则

判断某个版本（`trx_id`）对当前事务是否可见：

```
trx_id = creator_trx_id?               ──→ 是自己修改的，可见 ✅
      ↓ 否
trx_id < min_trx_id?                   ──→ 已提交的旧事务，可见 ✅
      ↓ 否
trx_id ≥ max_trx_id?                   ──→ 创建 ReadView 之后才启动的事务，不可见 ❌
      ↓ 否（在 [min_trx_id, max_trx_id) 之间）
trx_id 在 m_ids 列表中?                 ──→ 事务仍活跃，不可见 ❌
      ↓ 否
事务已提交，可见 ✅
```

#### 可见性规则速查表

| 条件 | 含义 | 可见性 |
|------|------|:------:|
| `trx_id == creator_trx_id` | 自己修改的版本 | ✅ |
| `trx_id < min_trx_id` | 修改事务已提交且比 ReadView 早 | ✅ |
| `trx_id >= max_trx_id` | 修改事务在 ReadView 之后才启动 | ❌ |
| `trx_id ∈ m_ids` | 修改事务仍活跃（未提交） | ❌ |
| `trx_id ∉ m_ids` | 修改事务已提交 | ✅ |

---

### 读提交 VS 可重复读

两种隔离级别的核心区别在于 **ReadView 的生成时机**：

| 隔离级别 | ReadView 生成时机 | 效果 |
|---------|-----------------|------|
| **读提交 (RC)** | **每次**执行 SELECT 语句前都生成一个新的 ReadView | 同一事务内多次读结果可能不同（不可重复读） |
| **可重复读 (RR)** | **事务内第一个** SELECT 语句生成 ReadView，之后**复用** | 整个事务看到的数据一致 ✅ |

#### 读提交示例

```
事务 B 执行流程：
① SELECT (生成 ReadView₁)     → 读到原始数据
② 事务 A 修改并提交
③ SELECT (生成 ReadView₂)     → 事务 A 已不在 m_ids，读到 A 修改后的数据
④ 前后两次结果不一致 → ❌ 不可重复读
```

#### 可重复读示例

```
事务 B 执行流程：
① SELECT (生成 ReadView，记录当前活跃事务快照)  → 读到原始数据
② 事务 A 修改并提交
③ SELECT (复用同一个 ReadView)                → ReadView 中事务 A 仍在 m_ids? 
                                                     ↓
                                              顺着 undo log 版本链往下找，
                                              直到找到 trx_id < min_trx_id 的旧版本 ✅
④ 前后两次结果一致 → ✅ 可重复读
```

---

### 幻读

#### 快照读：MVCC 解决了幻读

在可重复读隔离级别下，事务内所有普通 SELECT（快照读）复用同一个 ReadView，其他事务插入的新行对当前事务不可见，因此**查询结果集数量不变**。

```sql
-- 事务 A
SELECT COUNT(*) FROM t WHERE id > 10;  -- 返回 5 条
-- 此时事务 B 插入 id=11 并提交
SELECT COUNT(*) FROM t WHERE id > 10;  -- 复用 ReadView，仍返回 5 条 ✅
```

#### 当前读：Next-Key Lock 解决了幻读

**当前读**（`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`、`INSERT`）读取的是**已提交的最新数据**，MVCC 不生效，因此需要**加锁**防止幻读。

InnoDB 使用 **Next-Key Lock（记录锁 + 间隙锁）** 解决当前读的幻读问题：

| 锁类型 | 作用 |
|--------|------|
| **Record Lock**（记录锁） | 锁定**已存在的记录行** |
| **Gap Lock**（间隙锁） | 锁定两条记录之间的**间隙**，防止其他事务在间隙中插入新行 |
| **Next-Key Lock** | = Record Lock + Gap Lock，锁定**一个左开右闭区间 (a, b]** |

**示例**：执行 `SELECT * FROM t WHERE id > 10 FOR UPDATE`，InnoDB 会给 `(10, +∞)` 范围加上 Next-Key Lock，其他事务无法插入 `id > 10` 的新行，从而避免幻读。

**为什么当前读不能用 ReadView 防幻读？**

一句话：**当前读必须看到最新行，不能假装看不见。**

```
快照读：ReadView 过滤掉其他事务新插入的行——"假装没看见"
        → 幻读不存在了，不需要加锁 ✅

当前读：必须读到最新已提交的数据（否则修改会基于旧版本，导致更新丢失）
        → 能看见新插入的行，有幻读风险
        → 不能用 ReadView 来过滤（过滤了还怎么改？）
        → 只能加锁阻止别人插入：Next-Key Lock
```

简单说就是同一个问题两条路线：

```
防幻读的方法：
  ├─ 快照读：看见 → 但 ReadView 说"当我没看见"（过滤掉）
  └─ 当前读：看见 → 不能装瞎 → 锁住范围不让你插
```

#### 可重复读没有完全避免幻读

| 场景                               | 解决方式                     |       是否完全解决       |
| -------------------------------- | ------------------------ | :----------------: |
| **快照读**（普通 SELECT）               | MVCC 复用 ReadView         |       ✅ 完全解决       |
| **当前读**（SELECT ... FOR UPDATE 等） | Next-Key Lock（记录锁 + 间隙锁） |       ✅ 完全解决       |
| **先快照读，后当前读**                    | 当前读读到最新数据，可能看到之前插入的新行    | ⚠️ 特殊场景可能出现类似幻读的效果 |

```sql
-- 事务 A
BEGIN;
SELECT * FROM t WHERE age = 25;  -- 快照读：0 行（没有这条记录）

-- 事务 B 插入 age=25 并提交

SELECT * FROM t WHERE age = 25;  -- 快照读：仍是 0 行 ✓ MVCC 屏蔽了

UPDATE t SET name = 'X' WHERE age = 25;  -- 当前读，更新了 1 行！
-- 因为 UPDATE 是当前读，它看到了事务 B 插入的行

SELECT * FROM t WHERE age = 25;  -- 快照读：1 行 ⚠️ 出现了
-- 因为自己刚 UPDATE 过这行，trx_id 变成自己的，于是可见了
```

## 锁

### 全局锁

```sql
FLUSH TABLES WITH READ LOCK;  -- 整库只读
```

**作用**：整个数据库进入只读状态，其他线程的 DML、DDL 全部阻塞。

**应用场景**：全库逻辑备份（mysqldump），保证备份期间数据一致性。

---

### 表级锁

#### 表锁

```sql
LOCK TABLES t_student READ;   -- 当前会话只能读该表，其他会话可读不可写
LOCK TABLES t_student WRITE;  -- 当前会话可读写该表，其他会话不能读写
```

表锁是显式加的，锁住后当前会话不能访问其他表，需 `UNLOCK TABLES` 释放。

#### 元数据锁（MDL）

MDL 是 MySQL **自动加**的表级锁，保护表结构，防止读写数据时表结构被修改。

| 操作 | 加锁类型 | 说明 |
|------|---------|------|
| CRUD（SELECT/INSERT/UPDATE/DELETE） | **MDL 读锁** | 读读不阻塞 |
| DDL（ALTER TABLE 等结构变更） | **MDL 写锁** | 读写、写写均冲突 |

> MDL 写锁会与已有的 MDL 读锁冲突，如果大事务长时间持有 MDL 读锁，后续 DDL 会被阻塞，进而阻塞该表的所有后续查询。

#### 意向锁（Intention Lock）

**作用**：表级锁，快速判断表中是否已有行锁，避免逐行检查。

| 类型 | 含义 |
|------|------|
| **意向共享锁（IS）** | 事务准备对某些行加 S 锁 |
| **意向独占锁（IX）** | 事务准备对某些行加 X 锁 |

- 对某行加 S 锁前，先给表加 IS 锁
- 对某行加 X 锁前，先给表加 IX 锁
- **意向锁之间不互斥**（IS 和 IX 可以共存）
- 意向锁与表级 S/X 锁互斥（如 IX 与表级 X 锁互斥）

#### AUTO-INC 锁（自增锁）

InnoDB 为 `AUTO_INCREMENT` 字段分配自增值时使用的表级锁，**INSERT 语句执行完就释放**（不等事务提交）。

传统 AUTO-INC 锁模式下，多个事务并发插入需串行等待，性能差。

参数 `innodb_autoinc_lock_mode` 控制锁模式：

| 模式 | 名称 | 行为 | id 连续性 |
|------|------|------|:--------:|
| 0 | 传统模式 | 所有 INSERT 都持锁到语句结束 | 连续 |
| **1**（默认） | **连续模式** | 普通 INSERT 分配完立即释放；批量 INSERT 持锁到语句结束 | 连续 |
| 2 | 交错模式 | 所有 INSERT 都不持锁，交替拿 id | 可能不连续 |

**模式 1 的缺点**：当 `binlog_format = STATEMENT` 时，主库交错执行插入但 id 连续，从库按顺序回放 SQL 时分配的 id 顺序不同，导致主从数据不一致。**解决方法**：使用 `binlog_format = ROW`。
    

---

### 行级锁

#### Record Lock（记录锁）

锁定索引上的一条具体记录。

| 类型 | 含义 | 兼容性 |
|------|------|--------|
| **S 锁**（共享锁/读锁） | 只读不修改 | S + S 兼容，S + X 互斥 |
| **X 锁**（独占锁/写锁） | 修改数据 | 与任何锁都互斥 |

#### Gap Lock（间隙锁）

锁定两条记录之间的**间隙**（不锁定记录本身），防止其他事务在间隙中**插入新行**，解决幻读。

- **间隙锁之间互相兼容**（和 Record Lock 不同）
- 间隙锁是**前开后开区间** `(a, b)`

#### Next-Key Lock（临键锁）

= **Record Lock + Gap Lock**，锁定**前开后闭区间** `(a, b]`。

例如锁住 `(3, 5]`，其他事务不能插入 id=4 的记录，也不能修改/删除 id=5 的记录。

> X 型的 Next-Key Lock 之间互斥（因为记录锁部分互斥）。

#### 插入意向锁（Insert Intention Lock）

当事务试图插入记录时，如果目标间隙已被其他事务加了 Gap Lock / Next-Key Lock，则生成**插入意向锁**并进入等待状态。

- 名字带"意向"但**不是意向锁**，属于**行级锁**
- 本质是一种特殊的 Gap Lock，起**信号量**作用
- 持有间隙锁的事务提交后，插入意向锁被唤醒，事务获得插入机会

---

### 加锁规则

不同查询条件下，Next-Key Lock 的退化规则。

#### 唯一主键索引等值查询

| 查询结果       | 加锁方式                            | 原因                     |
| ---------- | ------------------------------- | ---------------------- |
| 记录 **存在**  | Next-Key Lock **→ Record Lock** | 唯一索引精确命中，无需间隙锁防幻读      |
| 记录 **不存在** | Next-Key Lock **→ Gap Lock**    | 第一条大于查询条件的记录上加间隙锁，防止插入 |

#### 唯一主键索引范围查询

| 查询条件 | 加锁行为 |
|---------|---------|
| `id >= 10` | 等值部分 `id=10` → Record Lock；范围部分 `id>10` → Next-Key Lock |
| `id <= 10`（10 存在） | 前面范围加 Next-Key Lock，`id=10` 处也是 Next-Key Lock |
| `id <= 10`（10 不存在） | 前面范围加 Next-Key Lock，第一个大于 10 的记录退化为 Gap Lock |
| `id < 10`（10 存在） | 前面范围加 Next-Key Lock，`id=10` 处退化为 Gap Lock |

#### 非唯一索引等值查询

| 查询结果 | 加锁行为 |
|---------|---------|
| 记录 **存在** | 扫描的二级索引记录加 Next-Key Lock；第一个不符合条件的退化为 Gap Lock；对应主键索引加 Record Lock |
| 记录 **不存在** | 第一个不符合条件的二级索引退化为 Gap Lock；不对主键索引加锁 |

> 非唯一索引等值查询不会退化为 Record Lock，因为非唯一索引可能有多个相同值，需要间隙锁防止幻读。

#### 非唯一索引范围查询

**Next-Key Lock 不会退化**。例如 `age >= 22` 范围查询，虽然 `age=22` 看起来是等值条件，但非唯一索引不具有唯一性，如果只加 Record Lock，只会锁住现有的age=22的所有记录，其他事务仍可插入 `age=22` 的新行导致幻读。
所以不退化，所有区间都是nextkey lock
如果是唯一索引会退化，因为这里唯一性会限制其他事务插入age=22的记录

## Undo Log

Undo Log 用于事务回滚，保证**原子性**。

- 事务执行过程中，修改数据前先记录 undo log
- 事务回滚时，根据 undo log 撤销已执行的操作：
  - **INSERT**：记录主键值，回滚时**删除**对应记录
  - **DELETE**：记录整行内容，回滚时**插入**回去
  - **UPDATE**：记录被更新列的旧值，回滚时**更新**回旧值

> undo log 也会写入 Buffer Pool 的 Undo 页面，因此 undo log 本身也需要 redo log 来保证持久性。

## 日志

### Buffer Pool 概述

Buffer Pool 是 InnoDB 的内存缓冲区，缓存数据页和索引页，减少磁盘 I/O。

- **读**：先从 Buffer Pool 读取，未命中则从磁盘加载到 Buffer Pool
- **写**：先修改 Buffer Pool 中的页（标记为脏页），由后台线程择机刷盘

### Redo Log

#### 为什么需要 Redo Log

1. **实现持久性（Crash-Safe）**：系统崩溃后，已提交的事务不丢失
2. **WAL（Write-Ahead Logging）**：先写日志，后写磁盘，将随机写变为顺序写，提升写入性能

**工作流程**：
```
更新 Buffer Pool（标记脏页） → 写入 redo log（顺序写） → 事务提交（redo log 落盘）
```

此时更新就算完成了，脏页后续由后台线程择机刷盘。

#### Redo Log vs Undo Log

| | Redo Log | Undo Log |
|--|---------|---------|
| 记录内容 | 修改**后**的数据值（物理日志） | 修改**前**的数据值（逻辑日志） |
| 作用 | 崩溃恢复，保证**持久性** | 事务回滚，保证**原子性** |
| 写入时机 | 事务执行中持续写入 | 事务执行中，修改前写入 |

#### 为什么 Redo Log 比直接写数据快

- Redo Log 是**顺序写**（追加），磁盘 I/O 效率高
- 数据写入是**随机写**（需定位到具体位置），磁盘 I/O 效率低

#### 刷盘时机

| 时机 | 说明 |
|------|------|
| MySQL 正常关闭 | 全部刷盘 |
| redo log buffer 超过容量一半 | 自动刷盘 |
| 后台线程每 1 秒 | 定时刷盘 |
| **事务提交时** | 由 `innodb_flush_log_at_trx_commit` 控制 |

参数 `innodb_flush_log_at_trx_commit`：

| 值 | 行为 | 安全性 |
|:-:|------|:------:|
| 0 | 留在 buffer，不刷盘 | 丢失最后一批日志 |
| **1**（默认） | **立即 fsync 到磁盘** | **不丢失** ✅ |
| 2 | 写入 Page Cache，不立即落盘 | 操作系统崩溃时可能丢失 |

#### Redo Log 的环形写入

Redo Log 采用固定大小的环形空间，循环写入：

```
write pos → 当前写入位置
checkpoint → 最早可覆盖的位置
```

- `write pos` 追上 `checkpoint` → redo log 满了 → MySQL 阻塞更新操作 → 强制刷脏页推进 `checkpoint`
- 空间释放后继续执行

### Binlog

#### Binlog vs Redo Log

| | Binlog | Redo Log |
|--|--------|---------|
| **层次** | Server 层，**所有存储引擎可用** | InnoDB 层，仅 InnoDB |
| **日志类型** | 逻辑日志（SQL 或行变更） | 物理日志（页的物理修改） |
| **写入方式** | **追加写**，文件满了建新文件，不覆盖 | **循环写**，空间固定，覆盖可复用部分 |
| **用途** | 主从复制、数据恢复、归档备份 | 崩溃恢复，保证事务持久性 |

#### 主从复制

| 模式 | 行为 | 特点 |
|------|------|------|
| **同步复制** | 主库等待**所有从库**复制完成才提交 | 数据最安全，性能最差 |
| **异步复制** | 主库提交时不等待从库确认 | 性能最好，可能丢数据 |
| **半同步复制**（推荐） | 主库等待**至少一个从库**确认收到 binlog | 兼顾性能和安全 |

实现读写分离：**写请求走主库，读请求走从库**，减轻主库压力。

#### 写入与刷盘

事务执行中，binlog 先写入 **binlog cache**。事务提交时一次性写入 binlog 文件，保证 binlog 的原子性。

参数 `sync_binlog` 控制刷盘频率：

| 值 | 行为 |
|:-:|------|
| 0 | 只 write，不主动 fsync，由操作系统决定刷盘时机 |
| **1**（最安全） | **每次提交都 write + fsync** |
| N | 每累计 N 个事务才 fsync 一次 |

#### 两阶段提交

解决 redo log 和 binlog **写顺序不一致导致的主从不一致**问题。

**两阶段提交流程**：
```
① prepare 阶段：redo log 写入，标记 prepare
② 写 binlog 并刷盘
③ commit 阶段：redo log 标记 commit
```

**异常恢复规则**：
- 宕机时若 redo log 为 **prepare** 状态 → 检查 binlog 是否完整
  - binlog **完整** → binlog 已写入，提交事务
  - binlog **不完整** → binlog 未写入，回滚事务
- 宕机时 redo log 为 **commit** 状态 → 事务已完整提交，直接恢复

### Buffer Pool 管理

InnoDB 通过三种链表管理缓存页：

| 链表 | 作用 |
|------|------|
| **Free List**（空闲链表） | 管理空闲页 |
| **Flush List**（脏页链表） | 管理脏页（等待刷盘） |
| **LRU List** | 管理脏页 + 干净页，淘汰不常用数据 |

#### 预读失效

MySQL 加载数据页时，会将其**相邻的数据页一并加载**（预读），以减少磁盘 I/O。但这些预读的页可能实际未被访问，导致白读。

#### Buffer Pool 污染

某个 SQL 扫描大量数据时，可能将 Buffer Pool 中的**所有热数据都淘汰出去**，等热数据再次被访问时又需从磁盘加载，导致性能急剧下降。

#### InnoDB 的 LRU 优化

InnoDB 对标准 LRU 做了两项优化：

1. **LRU 链表分为 Young 和 Old 两个区域**
   - 预读的页优先插入 Old 区域
   - 被访问后才进入 Young 区域
   - 解决预读失效问题

2. **Old 区域停留时间阈值（`innodb_old_blocks_time`，默认 1 秒）**
   - 页被访问且停留超过阈值后，才移入 Young 区域
   - 防止批量扫描将热数据全部淘汰

> 可通过 `innodb_old_blocks_pct` 调整 Young 和 Old 区域的比例。

> **性能抖动提示**：如果在慢 SQL 监控中发现"偶尔"出现用时稍长的 SQL，可能是脏页刷盘导致的性能抖动。频繁出现时可调大 Buffer Pool 空间或 redo log 大小。

## 补充知识

### JOIN 连接

| 类型 | 说明 |
|------|------|
| **INNER JOIN**（内连接） | 返回两个表中**有匹配关系**的行（交集） |
| **LEFT JOIN**（左外连接） | 保留左表所有行，右表无匹配则填充 NULL |
| **RIGHT JOIN**（右外连接） | 保留右表所有行，左表无匹配则填充 NULL |
| **CROSS JOIN**（交叉连接） | 返回笛卡尔积，左表每行 × 右表每行 |

### 三大范式

| 范式            | 核心要求                     | 通俗理解                            |
| ------------- | ------------------------ | ------------------------------- |
| **1NF**（第一范式） | 每列都是不可分割的基本数据单元          | 字段不能再拆（如地址拆为省市）                 |
| **2NF**（第二范式） | 每列都与**主键完全相关**，而非部分依赖    | 复合主键下不能只依赖其中一部分（如订单表中商品名称应放商品表） |
| **3NF**（第三范式） | 每列都**直接依赖主键**，而非依赖其他非主键列 | 不能通过中间人（如订单表存客户编号而非客户详情）        |
复合主键：(订单ID, 商品ID)，而商品名称字段只依赖于商品ID，不满足第二范式
而第三范式单个字段主键也可能违反
`客户姓名 → 客户公司 → 客户电话` 都依赖于客户编号，而客户编号依赖于主键（订单ID）
即：`订单ID → 客户编号 → 客户电话`
**客户电话**间接依赖主键（通过客户编号），违反 3NF

> 
> 如果主键只有一个字段，自动满足 2NF。2NF 只有在复合主键时才有意义。

| 类型 | 特点 | 最大长度 | 存储方式 |

### 查询语法

#### IN vs EXISTS

| | IN | EXISTS |
|--|----|--------|
| **执行顺序** | 子查询先执行，结果集作为集合，外层匹配 | 外层逐行遍历，判断子查询是否有返回 |
| **适用场景** | 子查询结果集**小** | 子查询结果集**大**，有短路机制，容易走索引 |
| **NULL 影响** | NOT IN 时子查询结果含 NULL → 整体条件失效 | EXISTS 只判有无记录，不受 NULL 影响 |

#### UNION vs UNION ALL

| | UNION | UNION ALL |
|--|-------|-----------|
| 是否去重 | ✅ 自动去重 | ❌ 所有结果合并 |
| 效率 | 较低（需排序去重） | 较高 |

#### COUNT

```sql
COUNT(*)     -- 统计所有行（包括 NULL），与 COUNT(1) 等价
COUNT(1)     -- 统计所有行（包括 NULL），比 COUNT(主键) 少一步读取字段值
COUNT(列名)  -- 只统计该列不为 NULL 的行数
```

> 优化器优先选择**二级索引**进行 COUNT 扫描（二级索引比主键索引小）。

#### 别名与 WHERE

```sql
SELECT salary * 2 AS total FROM emp WHERE total > 5000;  -- ❌ 报错
```

**原因**：SQL 执行顺序中 `WHERE` 在 `SELECT` **之前**执行，WHERE 执行时尚不知道别名。

> 完整执行顺序：`FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT`
