# JVM

## 一句话定义

**JVM（Java Virtual Machine）** 是 Java 程序的运行环境：它负责加载 `.class` 字节码、管理运行时内存、执行代码，并通过 **GC** 自动回收不再使用的对象。

## 为什么需要 JVM

JVM 解决的是「一次编译，到处运行」和「运行时资源管理」的问题。Java 源码先被编译成平台无关的 **字节码**，再由不同操作系统上的 JVM 解释或编译执行。这样 Java、Kotlin、Scala、Groovy 等能编译成字节码的语言，都可以运行在 JVM 上。

另外，JVM 还提供了 **自动内存管理**、**类加载机制**、**JIT 即时编译** 和 **运行时监控工具**。这些能力让业务代码不用手动释放对象内存，也能在运行中通过热点代码编译获得更高性能。

## 什么时候重点关注 JVM

- 程序出现 **OOM、频繁 GC、CPU 飙高、响应时间抖动** 时。
- 面试中被问到 Java 内存模型、GC、类加载、调优排查时。
- 服务进入高并发或大内存场景，需要理解对象分配、晋升和回收行为时。

## JVM 和 Java 进程

一个 Java 程序启动后，通常对应一个 **操作系统进程**，这个进程里运行着一个 JVM 实例。不是系统里单独常驻一个「JVM 进程」给所有 Java 程序共用，而是每个 Java 进程自己带着一套 JVM 运行时。

如果同一台机器上启动两个 Java 程序，一般就是两个 Java 进程，也就是两个 JVM 实例。启动一个 `HelloWorld` 时，编译阶段会运行 `javac` 进程，执行阶段会运行 `java` 进程。

## JVM 整体结构

| 模块 | 作用 |
|---|---|
| **类加载子系统** | 把 `.class` 文件加载到内存，生成 `Class` 对象 |
| **运行时数据区** | 保存对象、栈帧、类元数据、程序计数器等运行时数据 |
| **执行引擎** | 解释执行字节码，或通过 JIT 编译成机器码 |
| **垃圾回收器** | 回收不再可达的对象，释放堆内存 |
| **本地方法接口 JNI** | 调用 C/C++ 等 native 方法 |

JIT 会把热点代码编译后放入 **Code Cache**，后续执行时可以直接运行机器码，而不是反复解释字节码。

---

## JVM 内存区域

| 区域 | 线程共享 | 存放内容 | 常见问题 |
|---|---:|---|---|
| **程序计数器** | 否 | 当前线程正在执行的字节码行号 | 基本不会 OOM |
| **虚拟机栈** | 否 | 方法调用产生的栈帧、局部变量表、操作数栈 | `StackOverflowError` |
| **本地方法栈** | 否 | native 方法调用信息 | 栈溢出 |
| **堆** | 是 | 对象实例、数组 | `OutOfMemoryError: Java heap space` |
| **方法区 / 元空间** | 是 | 类元数据、常量、静态变量等 | `Metaspace` OOM |

### 局部变量表里一定有变量吗

如果是 **静态方法**，并且方法没有参数、没有局部变量，那么局部变量表可以没有业务变量。

如果是 **非静态方法**，即使方法体为空，局部变量表里也会隐式保存一个 `this` 引用。因为实例方法调用时，需要知道当前方法属于哪个对象。

```java
class Demo {
    static void a() {}
    void b() {}
}
```

- `a()` 是静态方法，没有 `this`。
- `b()` 是实例方法，局部变量表第 0 位通常是 `this`。

## 对象创建和内存布局

### 对象创建过程

1. **类加载检查**：确认对象所属类是否已经被加载。
2. **分配内存**：在堆中为对象分配空间，常见方式是指针碰撞或空闲列表。
3. **初始化零值**：对象字段先被赋默认值，比如 `int` 为 0，引用为 `null`。
4. **设置对象头**：写入 Mark Word、类型指针等运行时信息。
5. **执行构造方法**：执行 `<init>()`，把字段赋成代码里期望的值。

### HotSpot 对象内存布局

| 部分 | 作用 |
|---|---|
| **对象头** | 保存哈希码、锁状态、GC 年龄、类型指针等 |
| **实例数据** | 对象真正的字段内容 |
| **对齐填充** | 保证对象大小按 8 字节对齐 |

对象布局是 JVM 规范允许实现差异的地方。HotSpot 和 OpenJ9 的具体实现不完全一样。

### 对象一定分配在堆上吗

默认理解是「对象在堆上分配」。但 JVM 可能通过 **逃逸分析** 判断对象不会逃出当前方法，此时可以做 **栈上分配**、**标量替换** 或 **锁消除**。这类优化由 JVM 自动完成，业务代码一般不直接控制。

## 堆内存分区

传统分代模型中，堆可以分为：

```text
堆
├─ 新生代
│  ├─ Eden
│  ├─ Survivor From
│  └─ Survivor To
└─ 老年代
```

新对象通常先进入 **Eden 区**。Minor GC 后仍然存活的对象，会在两个 Survivor 区之间复制，并增加对象年龄。当年龄达到阈值，或者 Survivor 放不下时，对象会晋升到 **老年代**。

### 大对象如何处理

**大对象** 是指占用连续内存较大的对象，比如大数组、长字符串等：

```java
int[] array = new int[1_000_000];
String str = new String(new char[1_000_000]);
```

在部分收集器和参数组合下，大对象可能直接进入老年代。参数 `-XX:PretenureSizeThreshold` 可以影响这个行为，但它并不是所有收集器都支持，且默认通常不依赖它判断对象晋升。

在 **G1** 中，如果对象大小超过一个 Region 容量的 50%，会被认为是 **Humongous Object**，直接放入 Humongous Region。

## 垃圾回收 GC

### 判断对象是否存活

Java 主要使用 **可达性分析**，从一组 **GC Roots** 出发向下搜索引用链。能被访问到的对象是存活对象，访问不到的对象可以被回收。

| 方法 | 原理 | 问题 |
|---|---|---|
| **引用计数法** | 每个对象维护引用计数，为 0 就回收 | 解决不了循环引用 |
| **可达性分析** | 从 GC Roots 出发搜索引用链 | Java 主流方案 |

### GC Roots 有哪些

| GC Root | 说明 | 示例 |
|---|---|---|
| **栈帧局部变量** | 正在执行的方法里的对象引用 | `main()` 里的局部对象 |
| **静态变量** | 类级别变量，生命周期较长 | `private static List<?> list` |
| **常量引用** | 运行时常量池中的引用 | 字符串常量、Class 引用 |
| **JNI 引用** | native 方法中的引用 | JNI `NewGlobalRef` |
| **活跃线程** | 正在运行的线程对象 | `Thread.currentThread()` |
| **同步锁对象** | 被 `synchronized` 持有的对象 | `synchronized(obj)` |

JVM 不会每次 GC 都全量扫描所有内存。HotSpot 会借助 **OopMap** 记录栈和寄存器中哪些位置是对象引用；GC 发生时，线程到达 **Safe Point** 后，JVM 就能快速找到 Roots。

### 引用类型

| 类型 | 回收时机 | 典型用途 |
|---|---|---|
| **强引用** | 只要可达就不回收 | 普通 `new` 对象 |
| **软引用** | 内存不足前可能回收 | 缓存 |
| **弱引用** | 下次 GC 时回收 | `WeakHashMap`、`ThreadLocalMap` 的 key |
| **虚引用** | 不影响对象生命周期 | 跟踪对象回收，配合 `ReferenceQueue` |

### GC 算法

| 算法 | 过程 | 优缺点 |
|---|---|---|
| **标记-清除** | 标记存活对象，再清除未标记对象 | 简单，但产生内存碎片 |
| **标记-复制** | 把存活对象复制到另一块区域 | 无碎片，适合新生代，但浪费空间 |
| **标记-整理** | 标记后把存活对象向一端移动 | 无碎片，适合老年代，但移动成本高 |
| **分代收集** | 新生代和老年代使用不同策略 | 符合大多数对象朝生夕死的特点 |

### GC 类型

| 类型 | 触发场景 | 主要回收区域 |
|---|---|---|
| **Minor GC / Young GC** | 新生代空间不足 | 新生代 |
| **Major GC** | 老年代空间压力较大 | 通常指老年代 |
| **Mixed GC** | G1 中回收新生代和部分老年代 Region | 新生代 + 部分老年代 |
| **Full GC** | 堆或元空间压力大、显式触发等 | 通常全堆 + 方法区 |

### STW 和 Safe Point

**STW（Stop The World）** 是指 GC 或其他 JVM 操作发生时，暂停所有用户线程。暂停的目的是让对象引用关系在某个时间点保持一致，方便安全地移动或回收对象。

JVM 不会在任意机器指令处暂停线程，而是让线程运行到 **Safe Point**。常见 Safe Point 包括方法调用、循环回边、异常跳转等位置。

## 常见垃圾收集器

| 收集器 | 作用区域 | 特点 | 适用场景 |
|---|---|---|---|
| **Serial** | 新生代 | 单线程、复制算法 | 单核、小堆、客户端 |
| **ParNew** | 新生代 | Serial 的多线程版本 | 过去常搭配 CMS |
| **Parallel Scavenge** | 新生代 | 吞吐量优先 | 后台计算任务 |
| **Serial Old** | 老年代 | 单线程、标记整理 | Serial 搭配或兜底 |
| **Parallel Old** | 老年代 | 多线程、吞吐量优先 | 注重吞吐量 |
| **CMS** | 老年代 | 并发标记清除，低停顿 | 已逐步被淘汰 |
| **G1** | 全堆 | Region 化、可预测停顿 | JDK 9+ 默认，常用服务端 |
| **ZGC** | 全堆 | 超低停顿、大堆友好 | 大内存低延迟场景 |
| **Shenandoah** | 全堆 | 并发整理、低停顿 | 低延迟场景 |

### CMS 为什么被 G1 替代

CMS 的核心问题是使用 **标记-清除**，容易产生内存碎片。并发阶段还会占用业务线程的 CPU。如果老年代在 CMS 回收完成前被填满，就可能出现 **Concurrent Mode Failure**，退化成 Serial Old Full GC，停顿时间会明显变长。

### G1 的核心特点

G1 不再按连续的新生代、老年代来管理整个堆，而是把堆切成多个 **Region**。每个 Region 可以临时扮演 Eden、Survivor、Old 或 Humongous 区。

G1 的目标是「在可控停顿时间内，优先回收收益最高的 Region」。常见参数是：

```text
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200
```

它还通过 **Remembered Set** 记录跨 Region 引用，避免每次都扫描整个堆。

## 内存泄漏

### 一句话定义

**内存泄漏** 是指对象已经不再被业务使用，但仍然被 GC Roots 间接或直接引用，导致 GC 无法回收。

### 常见原因

| 原因 | 说明 | 处理方式 |
|---|---|---|
| **静态集合** | `static List` / `static Map` 只增不删 | 用完 `remove`，必要时用弱引用 |
| **资源未关闭** | 流、连接、Socket 未释放 | `try-with-resources` |
| **HashMap key 设计不当** | 放入后修改参与 `hashCode` 的字段，导致删不掉 | 避免可变字段参与哈希 |
| **非静态内部类** | 隐式持有外部类 `this` | 改成静态内部类 |
| **ThreadLocal 未清理** | 线程池复用线程，value 长期残留 | `finally` 中 `remove()` |
| **监听器未反注册** | `addListener` 后忘记 `removeListener` | 生命周期结束时反注册 |
| **类加载器泄漏** | 热部署后旧 ClassLoader 仍被引用 | 清理静态引用、线程、JDBC Driver 等 |

### ThreadLocal 为什么容易泄漏

`ThreadLocalMap` 的 key 是弱引用，但 value 是强引用。在线程池场景下，线程长期不销毁，如果没有调用 `ThreadLocal.remove()`，value 可能一直挂在线程上。

```java
try {
    threadLocal.set(value);
    // 业务逻辑
} finally {
    threadLocal.remove();
}
```

## OOM 和内存问题排查

### 处理过 OOM 怎么说

可以按这个思路回答：

> 我会先确认 OOM 类型，是堆溢出、元空间溢出，还是直接内存问题。然后保留现场，导出 Heap Dump，用 MAT 或 JProfiler 分析大对象和引用链。最后结合 GC 日志、业务代码和压测结果，判断是内存参数不足，还是代码存在泄漏。

```bash
jmap -dump:format=b,file=heap.hprof <pid>
```

分析重点：

1. 哪些对象占用最多内存。
2. 这些对象为什么还活着。
3. 引用链是否来自静态变量、线程、缓存、ThreadLocal 或 ClassLoader。
4. 修复后是否通过压测和 GC 日志验证。

### 栈溢出

**栈溢出** 通常发生在方法调用深度超过线程栈允许范围时，本质是新的栈帧无法继续分配。

常见原因：无限递归、递归层级过深、单个方法局部变量过多，或者线程栈参数 `-Xss` 设置过小。

## 类加载机制

### 类加载器

**类加载器** 负责把类的二进制字节流加载进 JVM，并生成对应的 `Class` 对象。

| 类加载器 | 作用 |
|---|---|
| **Bootstrap ClassLoader** | 加载 Java 核心类库 |
| **Extension / Platform ClassLoader** | 加载扩展类库或平台类库 |
| **Application ClassLoader** | 加载应用 classpath 下的类 |
| **自定义 ClassLoader** | 用于插件化、热部署、隔离加载等场景 |

### 类加载过程

类加载过程包括：**加载 → 验证 → 准备 → 解析 → 初始化**。

| 阶段 | 作用 |
|---|---|
| **加载** | 根据全限定类名获取字节流，转成方法区运行时数据结构，并生成 `Class` 对象 |
| **验证** | 检查 class 文件格式、语义和字节码安全性 |
| **准备** | 为静态变量分配内存并设置默认值 |
| **解析** | 把符号引用替换为直接引用 |
| **初始化** | 执行静态变量赋值和静态代码块，即 `<clinit>()` |

例子：

```java
static int a = 1;
```

在 **准备阶段**，`a` 的值是默认值 `0`；到 **初始化阶段**，才变成代码指定的 `1`。

![[Pasted image 20260512154811.png]]

### 双亲委派模型

**双亲委派模型** 是指类加载器收到类加载请求后，先把请求交给父加载器；父加载器无法加载时，子加载器才自己尝试加载。

为什么需要它：

1. **避免重复加载**：父加载器已经加载过的类，子加载器不用再加载。
2. **保护核心类库**：比如 `java.lang.String` 应该由 Bootstrap ClassLoader 加载，避免被用户自定义类篡改。

什么时候可能打破双亲委派：SPI 机制、Tomcat Web 应用隔离、OSGi、插件化、热部署。

## 常见线上问题排查

### CPU 占用过高怎么排查

1. 用 `top` 找到 CPU 高的 Java 进程。
2. 用 `top -Hp <pid>` 找到进程中 CPU 高的线程。
3. 把线程 id 转成十六进制。
4. 用 `jstack <pid>` 导出线程栈。
5. 在栈里搜索十六进制线程 id，定位具体方法。
6. 判断是死循环、频繁 GC、锁竞争，还是业务计算过重。

```bash
top
top -Hp <pid>
printf "%x\n" <tid>
jstack <pid> | grep -A 30 <hex_tid>
```

### 内存飙高怎么排查

1. 先用 `top` / `jstat` 看是不是 Java 堆在持续上涨。
2. 如果堆上涨明显，导出 Heap Dump。
3. 用 MAT / JProfiler 找大对象和 GC Roots 引用链。
4. 检查缓存、集合、ThreadLocal、文件上传、大对象、类加载器泄漏。
5. 修复后通过压测和 GC 日志确认曲线是否恢复正常。

### 频繁 Minor GC 怎么办

频繁 Minor GC 通常说明新生代对象创建太快，或者新生代空间太小。先看 GC 日志中 Young GC 的频率和耗时，再检查是否有大量短生命周期对象，必要时适当增大新生代或调整堆大小。

### 频繁 Full GC 怎么办

频繁 Full GC 通常说明老年代、元空间或直接内存有压力。重点看 Full GC 后老年代是否明显下降：如果下降明显，可能是堆太小或晋升太快；如果下降不明显，优先怀疑内存泄漏。

## JVM 调优

### 调优原则

JVM 调优不是先改参数，而是先明确目标：是降低平均响应时间、降低 P99 停顿、提高吞吐量，还是减少 OOM 风险。

调优流程：**监控 → 定位瓶颈 → 小步调整 → 压测验证 → 线上观察**。

![[Pasted image 20260512154622.png]]

### 常用参数

```bash
-Xms4g                            # 初始堆大小
-Xmx4g                            # 最大堆大小，通常和 Xms 设成一样，避免动态扩容抖动
-Xmn2g                            # 新生代大小
-Xss1m                            # 每个线程的栈大小
-XX:MetaspaceSize=256m            # 元空间初始触发 GC 阈值
-XX:MaxMetaspaceSize=512m         # 元空间最大值
-XX:+UseG1GC                      # 使用 G1
-XX:MaxGCPauseMillis=200          # G1 目标停顿时间
-XX:ParallelGCThreads=8           # 并行 GC 线程数
-Xlog:gc*:file=gc.log:time,uptime,level,tags   # JDK 9+ GC 日志
```

JDK 8 常见 GC 日志参数：

```bash
-XX:+PrintGCDetails -XX:+PrintGCDateStamps -Xloggc:gc.log
```

## 面试高频问题速记

### JVM 的价值是什么

JVM 屏蔽操作系统差异，让字节码跨平台运行；同时负责内存管理、垃圾回收、类加载和运行时优化。

### 堆和栈的区别是什么

栈是线程私有的，保存方法调用过程中的栈帧和局部变量；堆是线程共享的，主要保存对象实例。栈溢出通常是调用太深，堆溢出通常是对象太多或泄漏。

### Minor GC 和 Full GC 的区别是什么

Minor GC 主要回收新生代，频率高但通常较快。Full GC 通常涉及整个堆和方法区，停顿更长，需要重点关注。

### 为什么需要双亲委派

为了避免类重复加载，并保护 Java 核心类库不被用户自定义类替换。

### 内存泄漏和内存溢出的区别是什么

内存泄漏是不再使用的对象仍然被引用，导致无法回收；内存溢出是可用内存不足，无法继续分配对象。泄漏持续发展可能最终导致溢出。
