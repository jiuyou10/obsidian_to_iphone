# Obsidian 知识库工作规则

## 仓库结构
- `00-Inbox/` 草稿和待整理的笔记
- `10-Notes/` 永久笔记(原子化,一笔记一概念)
- `20-Topics/` 主题索引(MOC, Map of Content)
- `30-Sources/` 文献/资料出处
- `99-Archive/` 归档
- `_prompts/` 我自己用的 prompt 模板,不要修改

## 笔记规范
- 文件名:`概念名.md`,不加日期前缀
- 每篇笔记顶部用 YAML frontmatter:
```yaml
  ---
  tags: [领域/子领域]
  created: YYYY-MM-DD
  related: [[相关笔记]]
  ---
```
- 用 `[[双链]]` 引用其他笔记,不要用纯文本
- 永久笔记控制在 200-400 字,只讲一个概念
- 代码块标注语言:```python 而非 ```

## 写作风格
- 用我自己的话重述,不要照搬原文
- 概念解释先给「一句话定义」,再展开
- 技术笔记必须有「为什么需要它」和「什么时候用」两节
- 不堆砌术语,假设读者是三个月后的我自己

## 提纲式笔记填充规则
当看到笔记只有标题层级(# / ## / ###)而内容空缺,判定为「待填充提纲」。

填充风格:
- 每个 ### 小节结构:一句话定义 → 关键展开(2-4 句) → 必要时表格/代码 → 「为什么/何时用」
- 小节之间用 `---` 分隔
- 术语**加粗**,代码/类名用 `反引号`
- 不写"综上所述"、"接下来我们看"这类废话

填充工作流:
1. 先 cat 整篇提纲,理解脉络
2. 逐节填充前问我「这一节我打算这样写...,可以吗?」
3. 不确定的技术细节直接说"我不确定",不要编
4. 同一概念在不同地方出现,用 `[[双链]]` 指向主笔记

## Codex 行为约束
- 修改笔记前先 `cat` 显示原文,我确认后再改
- 创建新笔记前用 `grep -r` 搜索仓库,避免重复
- 不要自作主张归档或删除文件
- 大改动前提醒我先 `git commit` 做快照
